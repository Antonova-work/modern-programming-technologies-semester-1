# Вопрос 1
Что выведет этот код? Объясните порядок выполнения блоков.

class Demo {
static { System.out.print("A "); }
{ System.out.print("B "); }
Demo() { System.out.print("C "); }
public static void main(String[] args) {
System.out.print("D ");
new Demo();
System.out.print("E ");
new Demo();
}
}

**Ответ:** A D B C E B C

**Порядок выполнения:**
1. При загрузке класса выполняется статический блок — выводится "A"
2. Выполняется метод main — выводится "D"
3. Создание первого объекта:
    - Выполняется блок инициализации — выводится "B"
    - Выполняется конструктор — выводится "C"
4. Выводится "E"
5. Создание второго объекта:
    - Выполняется блок инициализации — выводится "B"
    - Выполняется конструктор — выводится "C"

# Вопрос 2
Класс Child находится в пакете b, а класс Parent — в пакете a. Child extends Parent. Какие члены Parent доступны в Child? А какие доступны в другом классе пакета b, который не наследует Parent?

**Ответ:**

**В классе Child (наследник):**
- public — доступны
- protected — доступны (через наследование)
- package-private (по умолчанию) — не доступны (разные пакеты)
- private — не доступны

**В другом классе пакета b (не наследнике):**
- public — доступны (через экземпляр)
- protected — не доступны (доступны только при наследовании)
- package-private — не доступны
- private — не доступны

# Вопрос 3
Можно ли создать sealed-класс, у которого все наследники — record? Какое преимущество это даёт при использовании в switch?

**Ответ:**

Да, можно создать sealed-класс (или интерфейс) с наследниками-record:

```java
sealed interface Shape permits Circle, Rectangle {}
record Circle(double radius) implements Shape {}
record Rectangle(double width, double height) implements Shape {}
```

Преимущество в switch: record-наследники позволяют использовать pattern matching и извлекать компоненты напрямую без явного вызова геттеров:

```java
Shape s = new Circle(5);
String result = switch(s) {
case Circle(double r) -> "Круг с радиусом " + r;
case Rectangle(double w, double h) -> "Прямоугольник " + w + "x" + h;
};
```

# Вопрос 4
Что выведет этот код? Объясните, что такое String Pool и как работает intern().

String a = "Hello";
String b = new String("Hello");
String c = b.intern();
System.out.println(a == b);
System.out.println(a == c);
System.out.println(b == c);

**Ответ:**
false
true
false


**Объяснение:**

- **String Pool** — это специальная область памяти в heap (до Java 7 в PermGen), где хранятся уникальные строковые литералы для экономии памяти и оптимизации.
- `String a = "Hello"` — литерал, создаётся в String Pool (или берётся существующий)
- `String b = new String("Hello")` — создаёт новый объект в heap (вне пула), даже если в пуле уже есть такая строка
- `intern()` — метод, который помещает строку в пул (если её там нет) и возвращает ссылку на строку из пула
- `String c = b.intern()` — возвращает ссылку на строку из пула (ту же, на которую ссылается `a`)

Поэтому:
- `a == b` — false (разные объекты: пул vs heap)
- `a == c` — true (обе ссылаются на объект в пуле)
- `b == c` — false (heap vs пул)

# Вопрос 5
Почему этот код не компилируется? Как исправить?

interface A { default void hello() { System.out.println("A"); } }
interface B { default void hello() { System.out.println("B"); } }
class C implements A, B {}

**Ответ:**

**Почему не компилируется:** возникает конфликт имён default-методов. Класс `C` наследует две реализации метода `hello()` с одинаковой сигнатурой от интерфейсов `A` и `B`, и компилятор не может выбрать, какую использовать.

**Как исправить:** необходимо переопределить метод в классе `C`:

```java
class C implements A, B {
    @Override
    public void hello() {
        A.super.hello(); // вызвать реализацию из A
        // или B.super.hello() - вызвать реализацию из B
        // или написать свою реализацию
    }
}
```



# Вопрос 6
В чём разница между StringBuilder и StringBuffer? Приведите конкретный сценарий, когда нужен именно StringBuilder.

**Ответ:**

**Разница:**
- `StringBuffer` — потокобезопасный, все методы синхронизированы, поэтому медленнее
- `StringBuilder` — не потокобезопасный, методы не синхронизированы, поэтому быстрее

**Сценарий для StringBuilder:** однопоточное приложение, где требуется множество операций модификации строк, например, построение сложного JSON или SQL запроса:

```java
public String buildInsertQuery(String table, Map<String, Object> data) {
    StringBuilder sb = new StringBuilder("INSERT INTO ");
    sb.append(table).append(" (");
    
    for (String column : data.keySet()) {
        sb.append(column).append(", ");
    }
    sb.setLength(sb.length() - 2); // удаляем последнюю запятую
    sb.append(") VALUES (");
    
    for (Object value : data.values()) {
        sb.append("'").append(value).append("', ");
    }
    sb.setLength(sb.length() - 2);
    sb.append(");");
    
    return sb.toString();
}
```

# Вопрос 7
Что выведет этот код? Объясните, что происходит с переменной x в лямбде.

int x = 10;
Runnable r = () -> System.out.println(x);
// x = 20; // что будет, если раскомментировать?
r.run();

**Ответ:**

Код выведет: `10`

Если раскомментировать строку `x = 20;` — возникнет ошибка компиляции.

**Объяснение:**
Лямбда-выражения могут захватывать переменные из внешней области видимости, но только те, которые являются **effectively final** (эффективно финальными) — то есть переменные, значение которых не изменяется после инициализации.

В данном случае:
- Переменная `x` инициализирована значением 10
- Лямбда захватывает эту переменную
- Если оставить `x = 20;` закомментированной, `x` остаётся effectively final
- Если раскомментировать присваивание, `x` перестаёт быть effectively final, и компилятор выдаёт ошибку

Это требование необходимо для гарантии того, что захваченное значение остаётся неизменным во время выполнения лямбды.

# Вопрос 8
Напишите пример лямбда-выражения, которое нельзя заменить на ссылку на метод. Объясните почему.

**Ответ:**

**Пример лямбды, которую нельзя заменить на ссылку на метод:**

```java
Function<String, String> processor = s -> {
    if (s == null || s.trim().isEmpty()) {
        return "некорректный ввод";
    }
    return s.trim().toUpperCase() + " (обработано)";
};
```

Почему нельзя заменить на ссылку на метод:

Ссылка на метод (ClassName::methodName) может заменить только лямбду, которая просто вызывает один существующий метод с переданными аргументами

Данная лямбда содержит:

Проверку условия (if)

Несколько операций (trim, toUpperCase, конкатенация)

Возврат разных значений в зависимости от условия

Такая логика не может быть выражена одним методом, на который можно сослаться



# Вопрос 9
Чем EnumMap быстрее HashMap для ключей-enum? Опишите внутреннюю реализацию в одном предложении.

**Ответ:**

**Одним предложением:** `EnumMap` внутренне реализован как обычный массив, индексированный порядковым номером enum-константы (`ordinal()`), что даёт доступ за O(1) без вычисления хеша и разрешения коллизий.

**Детальное объяснение:**

- `HashMap` вычисляет хеш-код ключа, затем определяет бакет, обрабатывает возможные коллизии — всё это требует времени
- `EnumMap` использует тот факт, что все возможные ключи известны заранее (константы enum)
- Внутри `EnumMap` содержит массив длиной, равной количеству констант в enum
- При операциях `put(key, value)` или `get(key)` используется `key.ordinal()` как индекс в массиве
- Нет необходимости в вычислении хеша, сравнении equals(), нет коллизий, нет rehashing

Это делает `EnumMap` чрезвычайно эффективным как по скорости, так и по памяти.

# Вопрос 10
Дан record Point(int x, int y) {}. Что сгенерирует компилятор автоматически? Можно ли добавить в record изменяемое поле? Можно ли наследоваться от record?

**Ответ:**

**Что генерирует компилятор автоматически:**
1. Канонический конструктор со всеми полями (`Point(int x, int y)`)
2. Геттеры с именами полей (методы `x()` и `y()`)
3. `equals()` — сравнивает все поля
4. `hashCode()` — вычисляется на основе всех полей
5. `toString()` — возвращает строку вида `Point[x=..., y=...]`

**Ответы на вопросы:**

- **Можно ли добавить изменяемое поле?** — Нет, все поля в record неявно являются `final`. Record по определению является иммутабельным (неизменяемым) типом данных. Можно добавлять только статические поля.

- **Можно ли наследоваться от record?** — Нет, record неявно объявляется как `final`, поэтому от него нельзя наследоваться. Также record не может наследовать другой класс (кроме `java.lang.Record`), но может реализовывать интерфейсы.

**Пример с интерфейсом:**
```java
interface Drawable { void draw(); }
record Point(int x, int y) implements Drawable {
    @Override
    public void draw() {
        System.out.println("Рисуем точку (" + x + ", " + y + ")");
    }
}
```
